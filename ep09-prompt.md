## 主 Prompt（直接复制）

```
做一个单文件 HTML 3D 魔方求解器，要求：

【硬约束】
1. 单个 HTML 文件，零外部依赖（无 CDN、无 import、无 Three.js / Babylon）
2. 双击 HTML 文件就能在浏览器跑
3. 渲染只能用 Canvas 2D，禁用 WebGL
4. 3D 数学全部手写：旋转矩阵 + 透视投影 + 深度排序（Painter's algorithm）
5. 文件大小不超过 250KB

【UI 要求】
1. 主画布：3D 透视画布（占左 2/3，至少 600×600）
   - 标准三阶魔方（3×3×3 = 27 小块，外层 26 块可见）
   - 6 面标准颜色：白(U) / 黄(D) / 红(R) / 橙(L) / 蓝(F) / 绿(B)
   - 每个小方块带黑色边框、可见间隙
   - 默认初始为还原状态

2. 用户交互：
   - 鼠标左键拖动空白区：旋转整体视角（围绕 X 轴和 Y 轴）
   - 滚轮：缩放
   - 鼠标点击 + 拖动魔方某一面：旋转该面（90° 动画过渡）
   - 键盘快捷键（标准记号）：
     U / D / L / R / F / B  顺时针旋转该面
     U' / D' / L' / R' / F' / B'（Shift + 字母）逆时针
     Y / X / Z  整体绕轴旋转

3. 控制面板（右侧 1/3）：
   - 「打乱」按钮：随机生成 20 步打乱序列（动画播放）
   - 「还原」按钮：自动求解（动画播放每一步）
   - 「重置」按钮：直接回到还原状态
   - 「步数」计数器：显示当前与还原状态距离的步数
   - 「状态」文字：「已还原 ✓」/「未还原（距 N 步）」
   - 操作历史栏：显示最近 10 步操作记号（如 R U R' U'）

【物理/算法要求】
1. 旋转必须正确：U 顺时针 = 顶层从上往下看顺时针 90°（不能搞反，错一面扣 5 分）
2. 颜色映射必须符合标准（白上黄下，红右橙左，蓝前绿后）
3. 自动求解必须真求解（不能记录打乱序列后反向播放）
   - 接受任何合法解：可以是 LBL 法 / Kociemba / IDA* / 朴素 BFS
   - 求解步数 ≤ 50 步即可（不要求最优）
4. 旋转动画必须流畅（≥ 30 fps）
5. 状态判定必须正确：6 面颜色全部统一才算还原

【加分项】
1. 鼠标拖动魔方某一面时，能识别用户拖动方向 → 自动判定旋转哪一层、顺时针还是逆时针
2. 求解过程显示当前算法阶段（如「白十字 → F2L → OLL → PLL」）
3. 还原后播放庆祝动画（粒子 / 闪光 / 颜色变化）
4. 5 个预设挑战：
   - 「随机打乱（10 步）」
   - 「随机打乱（20 步）」
   - 「白十字残局」
   - 「最后一层 OLL 残局」
   - 「最后一层 PLL 残局」
5. 中文 UI 标注（不是机器翻译的英文）
6. 高 DPI 适配（devicePixelRatio）

```

## Prompt 设计原理

### 为什么这个题不会重蹈 EP08 物理判错覆辙

| EP08 翻车点 | EP09 防御措施 |
|---|---|
| 主观物理直觉争议 | **魔方是离散状态机**：颜色要么对、要么不对，还原 ≡ 6 面颜色均一，用 `JSON.stringify(state)` 直接比较 |
| 作者认知偏差 | **魔方转动记号全世界统一**（U/D/L/R/F/B + '），Kociemba 、 ACube 都能验证 |
| 物理直觉错误难以发现 | **评分全是可计算项**：转动正确性 / 还原步数 / 状态判定准确率 |
| 评论区反扑 | **本期 Hook 鸣谢 @TheGodIsABOY**：评论出题 → 作者兑现 → 系列梗复读 |

### 为什么禁止 WebGL / Three.js

- 3D 魔方 + Canvas 2D = 手写项目强制展示 3D 数学能力（旋转矩阵 + 透视投影 + 深度排序）
- 禁用 Three.js 避免强模型「一句 import 完事」、弱模型「不会装包」
- 3D 手写是必要难度梯度（EP02 圆锥 / EP03 九连环 已验证可行）

### 为什么不允许「记号反播」偷个懒

- 仅记录打乱序列 → 反向播放 ≡ 伪求解，一眼就看出来
- 评分时会外部手动输入任意状态，要求模型求解（不是他自己打乱的）
- 真求解算法 = 需要 LBL/CFOP/Kociemba/IDA*/BFS 中任一种，不限定
- 检验方式：多个初始状态 → 调用「还原」 → 最终状态与初始还原状态是否一致

## 评分自动化测试样例

> 录制完后，把每个 AI 的 HTML 文件丢进自动测试

```python
# Test 1：转动正确性 · 给 18 个单步动作，判状态累加是否与标准一致
SINGLE_MOVES = [
    "R", "R'", "R2",  "L", "L'", "L2",
    "U", "U'", "U2",  "D", "D'", "D2",
    "F", "F'", "F2",  "B", "B'", "B2",
]
# 对每个动作后读取 6 面颜色，与 ACube/Kociemba 标准状态对比
# 状态以 54 个字符表示（每面 9 个，顺序 UFRDLB）

# Test 2：还原正确性 · 给 10 个随机 20 步打乱状态，调用还原 → 检验 6 面颜色是否统一
SCRAMBLES = [
    "R U R' U' F2 R F' L D2 B'",
    "D F2 R' U L2 B U' R D2 F",
    "L' B2 D' F R2 U L D B' R'",
    # ...共 10 个
]

# Test 3：还原步数（不要求最优，但还原一个 20 步打乱 ≤ 50 步才算及格）
# Test 4：状态判定（人工调一个面颜色差一点 → 应该识别「未还原」）
# Test 5：动画性能（录 5 秒还原动画 → 平均 fps 必须 ≥ 30）
```

测试方法：
1. 用 Playwright 加载 HTML
2. 模拟键盘输入 → 读取页面 6 面颜色状态
3. 与标准状态机（用 [cubejs](https://github.com/ldez/cubejs) 或 ACube 作为 ground truth）对比
4. 还原验证不仅要动画起来像还原、还要最终 6 面颜色状态是「完全还原」的 54 字符串

## 评分流程（录制完成后）

1. **绝对分**：按 5 维度评分表（3D 渲染 25 + 转动正确性 25 + 还原算法 25 + 交互 15 + UI/动画 10）逐项打分
2. **MiniMax 基准**：MiniMax 实际得分作为「100 分」
3. **主榜换算分** = 主榜实际分 ÷ MiniMax 实际分 × 100（四舍五入）
4. **公开数据**：同时展示实际分 + 换算分 + 等级，避免歧义
